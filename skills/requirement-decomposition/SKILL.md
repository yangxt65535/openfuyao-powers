# openFuyao 需求拆解工作流

将原始需求经过 IR→SR→AR 三级拆解，产出标准化需求基线文档。本技能**仅覆盖需求分析**，不涉及架构规划、方案设计与具体实现。

## When to Use

- 收到企业内部立项或外部社区伙伴的原始需求
- Feature Request / Issue 需要评估可行性
- 版本规划时需批量拆解需求
- 需求评审前需要输出结构化基线文档

## When NOT to Use

- 纯 Bug 修复（使用 issue-fixer 类技能）
- 已有完整 SR/AR 需要进行方案设计（使用 solution-design 类技能）
- 单行配置变更等琐碎改动

---

## 核心概念：IR → SR → AR

- IR (initial requirement 初始需求): 站在内外部客户/市场角度，以准确的语言（完整的背景、标准的格式）重新描述的需求。
- SR (System requirement 系统需求): 为支撑“IR（初始需求）”所需要支持的具体需求，是站在研发视角，描述系统对外呈现的、可测试的全部功能需求和非功能需求。其中功能需求是对系统提供的功能的场景化的具体要求，非功能需求是对系统的成本、全局质量属性、技术限制等非功能性方面的具体要求。
- AR (Allocation Requirement 分配需求): 根据基层组织分工不同，以可交付的视角，由“系统需求SR”进一步分解 分配到子系统/模块的功能或非功能需求。

IR->SR、SR->AR 是一对一或一对多的关系，每个 SR 只能关联一个 IR，每个 AR 只能关联一个 SR。层级关系如下表所示：
```
IR: <初始需求标题>
  ├─ SR1: <系统需求 1>
  │   ├─ AR1.1: <子系统 A 的交付项>
  │   ├─ AR1.2: <子系统 B 的交付项>
  |   └── ...
  └─ SR2: <系统需求 2>
  |   ├─ AR2.1: ...
  |   └─ AR2.2: ...
  └── ...

# 举例说明
IR: 智能路由新增容灾场景流量管理能力
  ├─ SR1: 通用容灾场景流量管理能力
  │   ├─ AR1.1: 开源网关 + 智能路由容灾设计案例
  │   └─ AR1.2: 故障自动切流、实例状态发现、基础负载均衡
  ├─ SR2: 网关插件 & 端到端集成部署两种交付形态
  │   ├─ AR2.1: 网关插件形态打包与发布
  │   └─ AR2.2: 端到端集成部署脚本与验证
  └─ SR3: ...
```

**关键区分**：
- IR→SR 是**视角转换**：从客户/市场语言 → 研发可测试语言
- SR→AR 是**分配拆解**：从系统整体 → 子系统/模块的可交付项
- IR/SR/AR 均属于**需求分析**，不涉及"怎么做"（方案设计）

**非功能 SR 分配**：按相关子系统拆多条 AR，交付范围仍写需求不写方案。

## 工作流（六阶段，支持小需求直达）

```
阶段 1  ──→  阶段 2  ──→  阶段 3  ──→  阶段 4  ──→  阶段 5  ──→  阶段 6
IR 结构化    场景价值    SR 拆解     AR 分配     技术风险    工作量预估
(客户视角)   (优先级)   (研发视角)  (可交付项)  (宏观断点)  (基线输出)
```

### 阶段 1：IR 结构化

**目标**：将非结构化原始输入转为标准化的初始需求描述。

**定义**：站在内外部客户/市场角度，以准确的语言（完整的背景、标准的格式）重新描述的需求。不是简单复述原文，而是补充背景、明确痛点、界定范围。

**操作**：分析、明确需求，输出 IR 卡片

**关键原则**：
- 信息充分时直接填充并推进，不做不必要的确认
- 模糊项集中为一轮提问（使用 AskQuestion 批量提问），避免多轮来回
- 无法推断的项标记 `[待确认: 假设 xxx]`

**定义**：站在客户/市场角度对原始输入做结构化重述（补背景、消歧义、划边界），不是原文复述。

**要点（检查清单）**：
- 背景写清驱动力（市场/技术/合规/社区）
- 「期望能力」写**是什么**，不写技术方案
- 「约束条件」只列需求侧硬边界（版本、离线、性能、合规等）
- 来源（Issue/邮件等）可追溯；模糊处 `[待确认: 假设 xxx]`
- 多个**独立**客户诉求拆多条 IR；同一诉求的多面留在一条 IR，SR 再拆

**输出**：按下文 **IR 拆解** 的 IR 卡片模板产出 IR 卡片。

```markdown
## IR: <需求标题>

| 字段 | 内容 |
|------|------|
| 需求来源 | 企业立项 / 社区伙伴 / SIG 提案 / Issue#xxx / 客户反馈 |
| 需求标题 | <一句话概括，动词+对象+目标> |
| 背景描述 | <当前现状是什么，为什么需要改变> |
| 痛点分析 | <用户/客户遇到的具体问题，可列多条> |
| 期望能力 | <用户希望达到的效果，描述"是什么"而非"怎么做"> |
| 受益角色 | <运维人员 / 开发者 / 平台管理员 / 最终用户> |
| 影响范围 | <涉及的产品/子系统/组件（粗粒度）> |
| 优先级建议 | P0-紧急 / P1-高 / P2-中 / P3-低 |
| 约束条件 | <版本约束、兼容性、合规、离线、性能等硬约束> |
```

### 阶段 2：场景价值分析

**目标**：从技术价值和商业价值两个维度评估，给出优先级与优势场景。

**操作**：根据下文的模板完成并输出价值评估。

**使用原则**：

- 先给方向，再补数据：初次可用高/中/低，后续再量化
- 数据可追溯：新增信息尽量注明来源（客户访谈、历史工单、竞品分析等）
- 结论可修订：随着事实补充，优先级允许动态调
- 客观估计：不应过分夸大或贬低某一项价值点。参考当前项目背景、业界趋势、领域知识进行判断

**当前项目场景加分提示**

- K8s/CNCF 生态标准对齐
- AI 云原生能力增强（如 GPU 调度、推理服务）
- 多集群/混合云统一管控
- 异构算力（CPU、NPU）统合与释放

**输出模板**：

```markdown
### 技术价值（先填高/中/低）
| 维度 | 结论 | 依据（可后补） |
|------|------|----------------|
| 架构演进贡献 | 高/中/低 | |
| 技术债务消减 | 高/中/低 | |
| 复用性与通用性 | 高/中/低 | |
| 社区生态对齐 | 高/中/低 | |

### 商业价值（先填高/中/低）
| 维度 | 结论 | 依据（可后补） |
|------|------|----------------|
| 客户覆盖面 | 高/中/低 | |
| 竞品差异化 | 高/中/低 | |
| 营收关联度 | 高/中/低 | |
| 合规驱动力 | 高/中/低 | |

### 当前结论（可迭代）
- 优先级：P0/P1/P2/P3
- 优势场景：<1-3 个>
- 当前建议：纳入版本 / 继续调研 / 延后
- 信息缺口：<仍需补充的关键事实>
```

### 阶段 3：SR 系统需求拆解

**目标**：将 IR 转化为研发视角的、可测试的功能需求与非功能需求。

**定义**：为支撑“IR（初始需求）”所需要支持的具体需求，是站在研发视角，描述系统对外呈现的、可测试的全部功能需求和非功能需求。其中功能需求是对系统提供的功能的场景化的具体要求，非功能需求是对系统的成本、全局质量属性、技术限制等非功能性方面的具体要求。

**操作**：先做需求规模判定，再决定是否执行 IR→SR 拆解，输出单条 SR 卡片或多条 SR 列表。

**规模判定（先做）**：

根据该需求的影响面、技术难点等评估需求的粒度，根据粒度的小或大，决定是否要进行 SR 拆解。
- 若 IR 仅包含单一能力点、影响面单一、验收标准单一，则视为**小粒度需求**，需求直接由单一 SR 描述覆盖，跳过拆解。
- 若需求规模大，如涉及一个或多个仓库的重大架构变更，需要开发或集成全新模块等，则执行视为**大粒度需求**，需要拆解为多条 SR 。

**关键原则**：
- SR 描述系统**对外呈现的行为**，不描述内部实现
- 每条 SR 必须有明确的验收标准（具体目标产出）

**输出**：按下文功能 SR 模板，输出 SR 卡片或列表

```markdown

- **SR 编号**：SR<N>
- **SR 标题**：<系统需求标题>
- **关联 IR**: <初始需求标题>
- **需求类别**：功能 / 非功能，如果是非功能，还要明确 DFX 类别（性能、安全、兼容性、可服务性、可测试性等）
- **描述**：<系统应提供什么能力或DFX，面向什么场景/角色>
- **目标规格**：
  - R<N>.1: <可度量的功能指标或行为描述>
  - R<N>.2: ...
- **验收标准**：
  - [ ] AC<N>.1: <Given-When-Then 或明确的可验证条件>
  - [ ] AC<N>.2: ...
- **优先级**：Must / Should / Could
```

**要点**：目标规格可度量；验收标准可执行验证；只写对外可观测行为；优先级 Must/Should/Could 与 IR 可追溯。

---

### 阶段 4：AR 分配需求拆解

**目标**：将 SR 以可交付视角分解分配到子系统/模块。

**定义**：根据基层组织分工不同，以可交付的视角，由“系统需求SR”进一步分解 分配到子系统/模块的功能或非功能需求。

**操作**：将各个 SR 拆分为 AR，并输出 AR 表格。

**关键原则**：
- AR 是**分配**（谁负责交付什么），不是**设计**（怎么实现）
- 每条 AR 挂靠一条明确的 SR，仅能挂载一条
- AR 粒度以"一个子系统/模块可独立交付和验证"为准
- 如当前项目代码可用，读取相关模块代码了解现有能力边界

**分配维度**（择一或组合）：子系统（控制面/数据面/CLI/UI）/ 模块或包 / 组件（Controller/Webhook/Agent）/ 责任团队。

**原则**：可独立交付与验证；每条 AR 只挂一条 SR；同一 SR 下 AR 合集完整覆盖且无方案设计越界；粒度在「单模块可验」与「勿细到函数级」之间。

**输出**：按下文 **AR 模板**，输出 AR 列表

```markdown
## AR 模板：

- **AR 编号**：AR<N>.<M>
- **AR 标题**：<分配需求标题>
- **关联 SR**: SR<N>
- **分配目标**：<子系统/模块/组件名称>
- **交付范围**：<该子系统/模块需要提供什么能力>
- **验收标准引用**：AC<N>.X, AC<N>.Y（引用 SR 中的验收标准）
- **前置依赖**：<依赖其他 AR 的编号，或"无">
```

### 阶段 5：关键技术风险识别

**目标**：从项目宏观角度识别技术断点、大型架构变动等系统性风险。

本阶段关注的是**项目级别的系统性风险**，而非具体实现方案的风险（后者属于方案设计阶段）。

**风险识别矩阵**：

```markdown
| 编号 | 风险描述 | 影响 | 概率 | 缓解措施 |
|------|---------|------|------|---------|
| R1 | <风险点> | 高/中/低 | 高/中/低 | <应对方案> |
```

**风险类别与识别检查清单**：

**技术断点**（当前能力无法覆盖，需要突破）：
- 是否依赖尚不存在或未成熟的技术组件
- 是否需要引入全新技术栈，团队无经验积累
- 是否涉及尚无社区成熟方案的问题域

**大型架构变动**（对现有系统的冲击）：
- 是否要求对核心数据模型/CRD 进行不兼容变更
- 是否要求跨多个子系统的协调变更
- 是否涉及基础框架/核心依赖的替换

**外部依赖风险**：
- 上游依赖 API 是否稳定/是否已发版
- 是否依赖特定硬件/平台能力（GPU、特殊内核版本等）
- 离线/信创环境下依赖是否可获得

**规模与复杂度风险**：
- 需求影响面是否覆盖多个子系统（>3 个）
- 是否存在严格的前后依赖链导致并行度低
- 变更量是否超出常规迭代容量

**合规与兼容性风险**：
- 是否涉及 License 冲突
- 是否需要保持向后兼容（API/CRD/配置文件）
- 是否有安全合规硬约束（等保、信创认证等）

**关键原则**：
- "需求侧缓解建议"应聚焦于需求本身的调整（如拆分需求、降低约束、分阶段交付），而非具体技术方案
- 至少识别 3 个关键风险项

### 阶段 6：工作量预估与基线输出

**目标**：按 SR/AR 粒度给出工作量预估，考虑AI辅助研发带来的效率提升，输出完整基线文档。

**操作**：根据下文评估原则与模板完成并输出工作量预估表格。

**说明**：

- 先范围后精确：优先给区间值和信心度
- 区分人力与 AI 辅助：避免把 AI 产出当作零成本
- 持续校准：每轮评审后可按实际进展修订

**快速模板**：

```markdown
### 总体预估
| 指标 | 当前值 | 备注 |
|------|--------|------|
| SR 数量 | N | 小粒度需求可标注 SR=IR |
| AR 数量 | M | |
| 总工时（人天） | X~Y | 区间值 |
| AI 可辅助占比 | Z% | 预估范围 |

### 分项预估（按 SR/AR）
| 项 | 描述 | 人力工时 | AI 辅助工时 | 信心度 |
|----|------|----------|-------------|--------|
| SR1/AR1.1 | ... | X~Yd | A~Bd | 高/中/低 |

### 风险与波动
- 主要波动因素：<外部依赖/跨团队协作/技术不确定性>
- 保守系数建议：<如 1.2x>
```

**当前项目工作量参考**

- CRD 字段与 Controller 逻辑：2-5 人天/字段组
- CLI 子命令新增：1-3 人天/命令
- 包级框架迁移：0.5-1 人天/100 处调用点
- 单测补充：AI 辅助可覆盖 60-80%，人工审核约 0.5 天/包
- 文档产出：AI 初稿 + 人工审校约 0.5 天/篇

**注意**：此处为需求级工时预估，具体实施排期（阶段划分、Story 分解、依赖关系）属于方案设计阶段。

## 基线文档输出格式

**输出文件命名规约**：`<描述>-requirements.md`，其中 `<描述>` 为需求的简短英文标识（如 `Mooncake-Local-Cache`）。

最终输出一份完整的 Markdown 文档：

```markdown
# <需求标题> — 需求基线文档

## 1. IR 初始需求
（阶段 1 输出：IR 卡片）

## 2. 场景价值分析
（阶段 2 输出：技术/商业价值矩阵 + 综合结论）

## 3. SR 系统需求
（阶段 3 输出：各功能/非功能 SR 的规格 + 验收标准）

## 4. AR 分配需求
（阶段 4 输出：各 AR 的子系统分配 + 交付范围 + 验收标准引用）

## 5. 关键技术风险
（阶段 5 输出：风险矩阵 + 需求侧缓解建议）

## 6. 工作量预估
（阶段 6 输出：SR/AR 粒度预估 + AI 辅助说明）

## 7. IR→SR→AR 追溯矩阵
| IR | SR | AR | 验收标准 |
|----|----|----|---------|

## 8. 修改日志
| 版本 | 日期 | 说明 |
|------|------|------|

## 9. 参考目录
```

## 交互策略：最小化轮次

1. **能推断则不问**：基于输入文本、代码库、领域知识主动推断
2. **批量澄清**：将所有待确认项合并为一次提问（AskQuestion 一次提出所有问题）
3. **给出默认值**：不确定项标注 `[待确认: 假设 xxx]`，让用户确认或修正
4. **渐进式输出**：先输出 IR + 场景价值（阶段 1-2），用户确认方向后再展开后续
5. **代码驱动分析**：如工作空间有项目代码，主动读取架构文档和关键模块

## 边界声明

本技能**仅产出需求基线文档**。以下内容不属于本技能范围：
- 方案设计（架构视图、模块设计、接口定义）
- 实施规划（Story 分解、阶段排期、依赖关系图）
- 代码实现

需求基线完成后，方案设计请使用 solution-design 技能。

## 质量门禁

- [ ] IR 完整：背景、痛点、期望能力、约束条件均已填写
- [ ] 已完成 IR→SR 规模判定；若小粒度需求，已明确记录 `SR = IR` 及原因
- [ ] 每条 SR 可测试：有明确的目标规格和验收标准
- [ ] 每条 AR 可交付：挂靠明确的 SR，分配到具体子系统/模块
- [ ] IR→SR→AR 追溯矩阵完整，无孤立项
- [ ] 技术风险至少识别 3 个关键项并有需求侧缓解建议
- [ ] 工作量预估区分了人力与 AI 辅助
- [ ] 无遗留的 `[待确认]` 项（已全部澄清或标注了合理假设）
- [ ] 文档中无方案设计内容（无"怎么实现"、无架构图、无接口定义）
