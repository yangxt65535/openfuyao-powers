# oFEP 转换器

将用户的方案设计转换为标准 oFEP (openFuyao Enhancement Proposal) 格式的提案文档。

## 核心原则

1. **完整性**：原设计文档中的所有内容都必须映射到 oFEP，不可遗漏
2. **4+1 视图保留**：如果原方案包含 4+1 视图（上下文视图、逻辑视图、开发视图、运行视图、部署视图），必须完整保留
3. **章节保留**：oFEP 模板中的所有章节都必须保留，无内容的章节可以留空但不删除
4. **内容优先**：优先从原方案提取内容填充，无对应内容时保留模板结构或留空

## 工作流程

1. **接收输入**: 读取用户提供的方案设计内容
2. **分析内容**: 理解方案的核心目标、动机、技术细节、架构视图
3. **映射字段**: 将方案内容映射到 oFEP 模板的各个章节，确保 4+1 视图完整映射
4. **生成文档**: 输出符合 oFEP 规范的 Markdown 文件

## 输入处理

用户可能以多种形式提供方案设计：
- 直接文本描述
- 已有文档（Word、Markdown、脑图等）
- 零散的需求和想法

**如果用户提供了文件**：
- 读取文件内容
- 分析文件结构和关键信息
- **特别关注 4+1 视图**：识别并提取上下文视图、逻辑视图、开发视图、运行视图、部署视图

**如果用户只提供了文字描述**：
- 根据描述内容构建 oFEP 结构
- 缺少的视图章节保留标题但可留空

## 输出格式

生成的 oFEP 文档必须包含以下结构：

### YAML 前置元数据（Frontmatter）

```yaml
---
title: 提案标题
ofep-number: NNNN
authors:
  - ""  # 留空待填写
owning-sig: ""
participating-sigs: []
status: provisional  # 默认为初步草案
creation-date: yyyy-mm-dd  # 使用当前日期或留空
reviewers:
  - TBD
approvers:
  - TBD
see-also: []
replaces: []
replaced-by: []
stage: alpha
latest-milestone: ""
milestone:
  alpha: ""
  beta: ""
  stable: ""
feature-gates: []
disable-supported: false
metrics: []
---
```

**留空规则**：
- 编号 (ofep-number): 留空或使用 NNNN 占位
- 作者 (authors): 留空列表
- SIG (owning-sig, participating-sigs): 留空
- 评审人/批准人: 使用 TBD
- 日期: 使用 yyyy-mm-dd 占位或当前日期
- 版本里程碑: 留空

### 文档正文结构

必须包含以下章节（按顺序），**不可删除任何章节**：

1. **发布签核清单** - 保留模板中的标准清单项，标记为未完成
2. **摘要 (Summary)** - 从方案提取核心概述
3. **动机 (Motivation)**
   - 目标 (Goals)
   - 非目标 (Non-Goals)
4. **提案 (Proposal)**
   - 用户故事（可选，如方案中有使用场景则填写）
   - 风险与缓解措施
5. **设计细节 (Design Details)** - **重点章节，必须完整**
   - 架构与设计前提
   - 特性描述（背景与目标、依赖组件、License）
   - **4+1 视图**（如果原方案有则必须完整保留）：
     - 上下文与用例视图 (Context and Use Case View)
     - 逻辑视图 (Logical View)
     - 开发视图 (Development View)
     - 运行视图/进程视图 (Runtime/Process View)
     - 部署视图 (Deployment View)
   - 关键设计决策表
   - 模块接口与配置约定
   - DFX 质量属性设计（性能、可靠性、安全、兼容性、可服务性、可测试性、易用性）
   - Story/任务分解与排期
   - 测试计划（保留模板结构）
   - 毕业标准（保留模板结构）
   - 升级/降级策略
   - 版本倾斜策略
6. **生产可用性审查 (Production Readiness Review)** - 保留模板问卷结构，无内容可简化
7. **实施历史 (Implementation History)** - 留空或填入当前日期
8. **缺点 (Drawbacks)** - 分析方案的潜在缺点，如原方案有则保留
9. **替代方案 (Alternatives)** - 如有则列出考虑过的其他方案
10. **所需基础设施 (Infrastructure Needed)** - 可选，如方案涉及则填写

### 4+1 视图处理

如果原设计文档包含架构视图（通常以 PlantUML、图表或章节形式出现），**必须完整保留**：

| 视图类型 | 常见标题 | oFEP 映射位置 |
|---------|---------|--------------|
| 上下文视图 | 上下文与用例视图、用例视图、Context View | 设计细节 → 上下文与用例视图 |
| 逻辑视图 | 逻辑视图、Logical View、组件关系图 | 设计细节 → 逻辑视图 |
| 开发视图 | 开发视图、Development View、代码结构 | 设计细节 → 开发视图 |
| 运行视图 | 运行视图、Runtime View、进程视图、时序图 | 设计细节 → 运行视图 |
| 部署视图 | 部署视图、Deployment View、物理部署 | 设计细节 → 部署视图 |

**视图保留要求**：
- **绘图代码**（PlantUML、Mermaid 等）：直接复制原代码块，不做转换
- **图片链接**（如 `![描述](图片路径)`）：保留原文件路径，不做处理
- 保留视图的说明文字
- 如原方案使用其他图表工具，保持原样，不做 PlantUML 转换

### DFX 质量属性处理

如果原设计文档包含 DFX 设计（质量属性），必须保留：

- **性能规格** - 响应时间、吞吐量、资源限制
- **可靠性设计** - 容错、降级、恢复策略
- **安全设计** - 认证、授权、数据保护
- **兼容性设计** - API 兼容、版本兼容、迁移策略
- **可服务性/运维设计** - 监控、日志、告警
- **可测试性设计** - 测试策略、Mock 方案
- **易用性设计** - 用户体验、错误提示

### 注释块处理

- 保留模板中带有 `<!--` 和 `-->` 的指导性注释
- 删除模板顶部的"开始使用"流程注释
- 保留各章节内的填写说明注释
- 原方案中的注释（如设计意图）保留

## 内容转换规则

| 用户方案内容 | oFEP 章节 |
|------------|----------|
| 问题背景、为什么需要这个功能 | 动机 (Motivation) |
| 功能描述、预期效果 | 摘要 (Summary) + 提案 (Proposal) |
| 使用场景、用户案例 | 用户故事 (User Stories) |
| 技术架构、实现方案 | 设计细节 (Design Details) |
| 4+1 视图（各种架构视图） | 设计细节 → 对应的视图章节 |
| 关键设计决策、决策表 | 设计细节 → 关键设计决策表 |
| DFX/质量属性设计 | 设计细节 → DFX 质量属性设计 |
| Story 分解、排期、里程碑 | 设计细节 → Story 分解与排期 |
| 风险评估、注意事项 | 风险与缓解措施 (Risks and Mitigations) |
| 上线计划、发布策略 | 毕业标准 (Graduation Criteria) |
| 兼容性考虑 | 升级/降级策略、版本倾斜策略 |
| 依赖组件、License | 设计细节 → 依赖组件 |

## 填写策略

**必须从用户方案提取填充的内容**（不可留空）：
- 标题（基于方案主题）
- 摘要（核心概述）
- 动机（为什么需要）
- 目标/非目标
- 提案的具体内容
- 风险与缓解措施
- 设计细节的技术描述
- **4+1 视图**（如原方案有）
- **关键设计决策表**（如原方案有）
- **DFX 质量属性**（如原方案有）
- **Story 分解与排期**（如原方案有）

**保留模板占位符的内容**：
- 发布签核清单项（使用 `- [ ]` 格式）
- 测试计划结构（先决条件、单元测试、集成测试、e2e）
- 毕业标准结构（Alpha/Beta/GA/弃用）
- 实施历史（留空待后续填写）

**可以留空的章节**（但不可删除）：
- 生产可用性审查问卷（如原方案无相关内容，保留问题结构但可简答或留空）
- 部分 PRR 子章节（如原方案无对应内容）

**需要根据方案推断填充的**：
- 缺点（分析方案可能存在的问题）
- 替代方案（如果用户提到则保留）

## 章节留空规范

以下章节如果与原方案无关，可以简化或留空，但必须保留标题：

1. **生产可用性审查问卷 (PRR)** - 保留问题结构，可简答或写"参见设计细节"
2. **部分测试计划子章节** - 如原方案无详细测试计划，保留标题写"待补充"
3. **所需基础设施** - 如原方案未提及，保留标题或写"无"

**不可留空的核心章节**：
- 摘要、动机、提案、风险与缓解措施
- 设计细节及其所有子章节（4+1 视图、关键设计决策、DFX 等）
- 缺点、替代方案（如原方案有）

## 输出要求

1. 使用中文输出（与模板一致）
2. 保持模板原有的 Markdown 格式和结构
3. 确保所有 YAML 缩进正确（使用 2 个空格）
4. 保留所有注释块中的指导说明
5. 在无法确定的信息处留空或使用占位符
6. 使用代码围栏包裹整个输出，方便用户复制
7. **完整性检查**：输出前确认原方案的所有内容都已映射，4+1 视图完整保留

## 示例

**输入**: "方案设计文档包含以下内容：
- 背景：当前日志不规范
- 4+1 视图：包含 PlantUML 的上下文、逻辑、开发、运行、部署视图
- DFX：性能、安全、可靠性设计
- Story 分解：S1-S4 四个迭代
- 风险：双栈过渡风险、配置错误风险"

**输出**: 完整的 oFEP 文档，其中：
- 标题基于方案主题
- 摘要、动机提取背景和问题
- 设计细节包含完整的 4+1 视图（PlantUML 代码块保留）
- DFX 质量属性设计完整章节
- Story 分解与排期详细表格
- 风险与缓解措施完整表格
- PRR 问卷保留结构但可简化
