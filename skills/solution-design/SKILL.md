# openFuyao 方案设计工作流

基于需求基线文档，通过交互式探索产出标准化方案设计文档，包含 4+1 架构视图（PlantUML）、DFX 质量属性设计与可执行的工程实施方案。

## 硬性门禁：先比选、再成文

**阶段 2（多方案比选 / 多决策点拍板）与阶段 3（分块设计）之间必须有明确的用户决策；不得跳过。**  
若存在**多个彼此可独立选择**的问题（见下文「决策点解耦」），须**分别决策**，不得默认只沿一个维度选「整包方案」就把其它维度带过去。

| 顺序 | 允许产出 | 禁止产出（同一轮对话中） |
|------|----------|--------------------------|
| 阶段 2 单次回复 | **模式甲**：2-3 套**端到端**备选方案（统一结构）+ 权衡矩阵 + 可选推荐；**模式乙**：**决策点清单 Q1…Qn**（每项 2-3 选项与小权衡）+ 可选推荐 | 完整方案设计文档、块 1～5 任意正文、`# 特性设计` 起笔的长文档、4+1 PlantUML 全套、DFX/Story 成稿 |
| 用户回复后 | **全部决策点均有明确答复后**，进入阶段 3，从块 1 起按块输出 | 在用户未表态前「默认替你选」并写完整文档；**只答复了部分 Q 就进入阶段 3** |

**阶段 2 结束时的固定收尾句式**（须包含下列意图，可略改用语）：

- **模式甲**（整包比选）：请回复选用哪一方案、是否采纳推荐、是否混合或新增约束；**收到明确答复后再从块 1 起写正式文档**。
- **模式乙**（分项决策）：请对 **Q1…Qn 逐项**回复选项（如「Q1:B，Q2:A…」）；若某条需讨论，可只答该项并说明待定。**所有 Q 均有结论后再从块 1 起写正式文档**。
- **停止生成**：本回合在收尾句之后结束，不追加任何设计文档章节。

若用户在一开始就写明「已选定方案 X / 只写这一种」，可记录该决策并**仍在对话中**用简短文字复述权衡要点供确认一句，用户确认后再进入阶段 3（避免未对齐假设）。**若实际仍含多个独立决策点，不得因「已定总方案」而省略分项确认。**

### 成稿边界（与阶段 2 的区分）

- **阶段 2 产出**（对话中）：备选方案、权衡矩阵、Q1…Qn 选项、用户回复与 Agent 复述——**仅存在于交互过程**，不复制进最终 Markdown。
- **阶段 3 成稿**：只写**已定案**架构与决策；**不得**包含：未采纳方案详述、比选矩阵、选型记录专章、「用户选择方案 X」类表述。
- **关键设计决策表**（成稿内）：只列**已定结论**及理由，**不列**未采纳备选（见块 2 模板）。

### 决策点解耦（何时分项、禁止硬绑）

- **先列决策点**：进入阶段 2 时，用简短列表写出 **Q1、Q2…**（或沿用 D1/D2 编号），每个是一条**可单独回答**的设计选择（例：同步策略、API 版本策略、存储选型、观测方案），并注明**与哪些 Q 正交**、**哪些 Q 存在依赖**（依赖项说明「选 Q1=A 时 Q3 仅 B/C 有效」即可）。
- **禁止**：把多个正交问题塞进「方案 A / 方案 B」的打包叙述，使用户无法对某一维度改选而不误伤其它维度。
- **允许**：在**强耦合**时提供端到端「方案 A/B」（每条方案注明各维度取值组合）；同时**仍应用决策点表**列出各维度，便于用户改某一维时看清组合变化。
- **交互节奏**（二选一，在阶段 2 收尾中说明）：
  - **批量分项**：本回合列出全部 Q，请用户一次回复各 Q 选项（适合 Q 数量少、边界清晰）。
  - **逐项停等**：一次只呈现**一个** Q 的选项与权衡，用户回复后再给下一 Q（适合争议大、依赖多或 Q 较多时，**对齐阶段 1 的一问一答节奏**）。

## When to Use

- 需求分析（IR→SR→AR）已完成，进入方案设计阶段
- 需要输出社区提案或 SIG 提案文档
- 架构评审前需要编写设计文档
- 对已有方案进行重构/演进设计

## When NOT to Use

- 需求尚未明确（先用 requirement-decomposition 完成需求拆解）
- 纯 Bug 修复或配置变更（直接编码）
- 纯代码评审（使用 code-review 类技能）

## 核心设计原则

### 第一性原则驱动

从原始设计动机出发，而非从现有实现出发：

- 先问"这个问题的本质是什么"，再看"现有系统如何解决"
- 避免路径依赖——不因"改动小"而选择劣质方案
- 每个关键决策记录 **决策点编号（D1/D2/...）**、选定方案与理由

### 领域 SOTA 注入

主动搜索并引入领域最佳实践：

- 使用 WebSearch 工具查找 K8s/CNCF 社区同类问题的解决方案
- 对比 2-3 个业界方案（如 Cluster API、KubeVela、Crossplane 的做法）
- 形成差异化：明确指出"我们的方案在 X 维度优于 Y 方案，因为 Z"

### 架构自评估

对关键设计目标进行量化自评：

```markdown
| 设计目标 | 目标值 | 本方案预期 | 评估依据 |
|---------|-------|-----------|---------|
| <来自 SR 的关键指标> | <目标> | <预期> | <依据> |
```

## 设计工作流（四阶段）

**核心交互原则**：不直接从需求跳到完整设计。通过递进式交互逐步收敛方案。

```
阶段 1: 澄清 ──→ 阶段 2: 探索 ──→ 【用户决策】──→ 阶段 3: 设计 ──→ 阶段 4: 评审 ──→ 阶段 5: oFEP 转换
 (理解意图)   (比选/分项拍板)   （可含多轮）      (分块输出)      (自评+交付)      (生成提案)
```

`【用户决策】` 为必经节点：**每个待决 Q 均有结论前**不进入阶段 3；可多轮对话完成。

### 阶段 1：澄清与上下文建立

**目标**：深入理解设计意图、约束条件和成功标准。

**操作步骤**：

1. 读取需求基线文档，提取关键场景和验收标准
2. 如果工作空间有项目代码，读取 `CLAUDE.md`/架构文档/关键模块
3. 使用 WebSearch 查找领域 SOTA 参考
4. 向用户逐一提出澄清问题

**澄清问题框架**（一次提一个，按优先级排序）：

1. **目的确认**："需求文档提到 X，方案设计需要重点解决的核心问题是？"
2. **约束边界**："是否有以下硬约束？（兼容性版本、离线部署、多架构、性能指标）"
3. **成功标准**："方案评审通过的关键标准是什么？（参考 SR 验收标准确认）"
4. **技术偏好**："对 X 问题，倾向于方案 A（描述）还是方案 B（描述）？"
5. **提案模板**："是否有社区/SIG 指定的提案模板？如果有请提供。"

**关键原则**：

- 每次只问一个问题，等用户回答后再问下一个
- 前 2-3 个问题聚焦于高影响决策，低影响项可在设计中给出默认选择
- 当用户回答足以收敛方向时，停止提问，进入阶段 2

---

### 阶段 2：多方案探索与比选

**目标**：对关键设计选择给出可选路径与权衡，**分项或整包**均可，但必须**让用户逐项或整包明确表态**后再写正式文档；**本阶段（及为完成全部 Q 所需的后续回合）结束时停止，等待决策齐备**。

**操作步骤**：

1. **解耦决策点**：列出 Q1…Qn（见上文「决策点解耦」）；判断用**模式乙（分项）**、**模式甲（端到端方案）**或两者结合（端到端方案 + 决策点表）。
2. **模式甲**：基于澄清与 SOTA，设计 **至少 2 种**、通常 **2-3 种**端到端备选（若客观上只有一种可行路径，须说明原因，并对仍存在的**实现变体/风险档**分项让用户选）。
3. **模式乙**：对每个 Qk 给出 **至少 2 个**选项及优劣势（可用小表或短列表）；标注与别条的依赖。
4. 给出整体或分项的权衡矩阵；可有 **Agent 推荐**，并标注「最终以你的选择为准」。
5. 使用 **「阶段 2 结束时的固定收尾句式」**（批量分项或逐项停等，二选一已在收尾中声明），**不输出**阶段 3 及以后的文档正文。

**本阶段唯一交付物**：决策点表 / 方案对比 + 决策请求。**不得**在同一次助手回复中拼接块 1～5 或完整设计文档。

**决策点清单模板**（模式乙或组合模式时使用）：

```markdown
## 待决策项（正交维度）
| 编号 | 决策问题 | 与其它项关系 | 选项摘要 |
|------|----------|--------------|----------|
| Q1 | <一句话问题> | 与 Q2 正交 / 或：依赖 Q2 | A: … / B: … / C: … |
| Q2 | … | … | … |
```

**方案描述结构**：

```markdown
### 方案 A：<方案名称>

**核心思路**：<一句话概括>

**架构概要**：
<文本架构图或简要说明>

**优势**：
- ...

**劣势/风险**：
- ...

**适用场景**：<最适合在什么条件下选用>

**对应决策组合**（与 Q1…Qn 表联用时必填）：Q1=<取值>，Q2=<取值>，…
```

**权衡对比矩阵**：

```markdown
| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 实现复杂度 | 低/中/高 | | |
| 性能影响 | | | |
| 兼容性风险 | | | |
| 可扩展性 | | | |
| 社区对齐度 | | | |
| 工时预估 | | | |

**推荐**：方案 X。理由：...
```

**用户决策的可接受回复示例**（**仅在对话中**使用，**不写入**正式设计文档）：

- 模式甲：「采用方案 B」「按推荐执行」「混合：方案 A 的数据面 + 方案 B 的控制面」
- 模式乙：「Q1:B，Q2:A，Q3:C」或分多轮逐条回复直至全部 Q 有结论
- 「都不合适，补充约束：…」→ 回到阶段 2 修订选项，**仍不得在修订前写阶段 3 正文**

**收到决策后**：在**对话中**逐条复述用户在每个 Q（或整包方案）上的选择及附加约束，**确认无遗漏 Q** 后再开始阶段 3 块 1；成稿从块 1 起只写**结论态**设计。

---

### 阶段 3：分块设计与增量交付

**进入条件（缺一不可）**：

- 阶段 2 已输出比选/决策点材料，且用户已对**每一个**已列出的决策点给出结论（整包方案即视为各维度组合已一次性选定）；或用户预先声明选型且 Agent 已分项复述并经用户确认。
- **禁止**在仅完成阶段 1、阶段 2 无回复、或**仍有 Q 未决**的情况下，直接输出正式设计文档。

**目标**：按模块/复杂度分块输出设计，每块完成后征求用户意见。

**分块策略**（按以下顺序逐块输出）：

```
块 1: 特性描述 + 上下文视图 ──→ 征求意见
块 2: 4+1 架构视图           ──→ 征求意见
块 3: DFX 质量属性设计       ──→ 征求意见
块 4: Story 分解 + 排期      ──→ 征求意见
块 5: 总装 + 修改日志/参考    ──→ 最终确认
```

**每块输出后的标准交互**：

- 总结本块要点（3-5 句话）
- 明确请用户确认或提出修改意见
- 收到确认后才进入下一块

#### 块 1：特性描述与上下文

```markdown
# 1. 特性描述
<特性的业务描述，引用需求基线>

## 1.1 依赖组件
| 组件 | 描述 | 可获得性 |
|------|------|---------|

## 1.2 License
<与项目一致，说明新引入依赖的 License 兼容性>
```

#### 块 2：4+1 架构视图

为每个视图生成 PlantUML 图。详细模板见 [plantuml-templates.md](plantuml-templates.md)。


| 视图           | 关注点         | PlantUML 图类型         |
| ------------ | ----------- | -------------------- |
| **用例视图**（+1） | 核心参与者与用例场景  | `@startuml` Use Case |
| **逻辑视图**     | 模块/包结构与职责   | Component / Package  |
| **开发视图**     | 代码组织、包依赖、分层 | Package / Class      |
| **运行视图**     | 关键流程的运行时交互  | Sequence / Activity  |
| **部署视图**     | 节点、容器、网络拓扑  | Deployment           |


**设计要求**：

- 每个视图必须有 PlantUML 代码块（`@startuml` ... `@enduml`）
- 每个视图配 3-5 句文字说明设计意图
- 关键设计决策标注 D 编号（D1/D2/...），在决策表中记录

**PlantUML 渲染与存储规范**：

1. **检查 PlantUML MCP 可用性**
   - 首先确认是否配置了 `plantuml` MCP 服务器
   - 若 MCP 可用，进入步骤 2；若不可用，跳到步骤 5

2. **调用 MCP 生成 SVG**
   - 使用 `plantuml__generate_plantuml_diagram` 工具渲染每个 PlantUML 图
   - 参数：传入 `plantuml_code` 和 `format: "svg"`
   - **必须检查 MCP 返回的错误信息**，若有语法错误，需修改 PlantUML 代码直至正确

3. **保存 SVG 文件**
   - 在文档所在目录下创建 `figures/` 文件夹（如果不存在）
   - 将生成的 SVG 保存为 `figures/<view-name>.svg`（例如：`figures/use-case-view.svg`）
   - 文件名与视图对应：用例视图(use-case-view)、逻辑视图(logical-view)、开发视图(development-view)、运行视图(runtime-view)、部署视图(deployment-view)

4. **文档中引用 SVG 并保留源码**
   - 在文档中优先使用 SVG 文件路径引用图片
   - 同时使用 HTML `<details>` 标签折叠保留原始 PlantUML 代码块，便于后续编辑
   - 示例格式：
   ```markdown
   ![用例视图](figures/use-case-view.svg)

   <details>
   <summary>点击查看 PlantUML 源码</summary>

   ```plantuml
   @startuml
   ...
   @enduml
   ```

   </details>
   ```

5. **无 MCP 时的处理**
   - 如果 PlantUML MCP 不可用，**禁止**使用其他工具（如命令行 plantuml）强行生成 SVG
   - **禁止**直接生成 SVG 文件
   - 仅在文档中以代码块形式保留 PlantUML 源码，不引用外部图片
   - 示例格式：
   ```markdown
   ```plantuml
   @startuml
   ...
   @enduml
   ```
   ```

**关键设计决策表**（成稿：仅已定案，不写未采纳项）：

```markdown
| 编号 | 决策点 | 已定结论 | 理由 |
|------|--------|----------|------|
| D1 | ... | ... | ... |
```

#### 块 3：DFX 质量属性设计

覆盖以下质量属性（按需求基线中的规格对齐）：

```markdown
## 3.1 性能规格
| 规格名称 | 规格指标 |
|---------|---------|

## 3.2 可靠性设计
<故障恢复、降级策略、数据一致性>

## 3.3 安全设计
<认证鉴权、数据脱敏、配置安全、威胁建模（STRIDE，可选）>

## 3.4 兼容性设计
<API 兼容、CRD 版本策略、升级/回滚>

## 3.5 可服务性设计
<日志、监控、诊断、运行时配置>

## 3.6 可测试性设计
<单测策略、集成测试、Mock 方案>

## 3.7 易用性设计
<CLI UX、错误提示、文档、默认值>
```

**DFX 完备性检查清单**：

- 安全：是否考虑了敏感信息泄露（密码/token/证书）
- 安全：是否考虑了配置注入、路径穿越
- 可靠性：组件故障时的降级行为是否定义
- 兼容性：CRD/API 变更是否向后兼容
- 可测试性：关键路径是否可 Mock/可注入
- 易用性：错误信息是否对用户有指导意义
- 性能：热路径是否避免同步阻塞和日志刷屏
- 离线：离线环境下所有依赖是否可获得

#### 块 4：Story 分解与排期

**必须**与输入的**需求分析基线文档**中的 **SR、AR 编号及表述一致**（逐条 Story 挂靠，禁止自编编号或漂移表述）；用于实现与验收追溯。

每条 Story 除概述外，须给出**可照着做的开发指引**（具体操作），便于研发直接执行。

```markdown
## Story 设计与开发指引

### S1: <Story 名称>
- **关联 SR**：SRx（抄录基线中的标题或编号，与基线一致）
- **关联 AR**：ARy.z（同上，须与基线一致；一条 Story 可对应多条 AR，逐条列出）
- **目标摘要**：一句话说明本 Story 交付什么
- **验收标准**：列出基线 SR 中的 AC 编号及可验证条件（可与基线原文一致或引用）
- **接口/对外契约**（若有）：新增或变更的 API、CRD 字段、CLI、配置项名

#### 开发与验证操作清单（本 Story）
按执行顺序写清**具体操作**，避免只写原则：

1. **代码/配置**：拟修改的包路径或文件类型（如 `api/v1/*`、`internal/controller/*`、`config/crd/bases/*`）；关键逻辑要点（非大段代码）
2. **命令**：如 `make manifests` / `make test` / `go test ./...` / 本地 `kubectl apply` 验证步骤等（按项目惯例填写）
3. **数据与兼容**：迁移/升级步骤、Feature Gate、默认值、是否需 API 版本化
4. **验证**：如何确认本 Story 完成（对接哪条 AC；最小复现或检查命令）

### SR/AR → Story 追溯表（必填）
| Story | 关联 SR | 关联 AR | 备注 |
|-------|---------|---------|------|
| S1 | SR… | AR… | |
| S2 | … | … | |

### 实施排期
| 阶段/迭代 | Story | 关联 SR | 关联 AR | 主要操作摘要 | 预计工时 | 前置依赖 |
|-----------|-------|---------|---------|-------------|---------|---------|
```

**原则**：

- 基线中每条**本特性相关的 AR** 至少映射到一条 Story 或明确并入某 Story 的操作清单；若有 AR 不在范围内，在文内**显式说明**排除原因。
- 「开发与验证操作清单」须具体到**下一步可执行动作**；若依赖未知仓库布局，先根据工作空间补齐路径再写。

#### 块 5：总装与收尾

- 整合各块为完整文档；**剔除**任何误入成稿的阶段 2 比选材料（备选方案、矩阵、选型记录）
- 补充修改日志、参考目录
- 最终质量门禁检查

## 方案自评估

| 评估维度 | 评分(1-5) | 说明 |
|---------|-----------|------|
| 需求覆盖度 | | 所有 Must 级 SR 是否有对应设计 |

- oFEP 提案文档已生成（阶段 5，除非用户明确跳过）

### 阶段 5：oFEP 转换

**触发条件**：阶段 4 完成，方案设计文档已定稿并保存。

**目标**：基于已完成的方案设计文档，额外生成一篇符合 oFEP (openFuyao Enhancement Proposal) 规范的社区提案文档，用于社区评审与提案流程。

**操作步骤**：

1. **确认设计文档已保存**：阶段 4 产出的完整方案设计文档（含 4+1 视图 PlantUML、DFX 设计、Story 分解等）已定稿并写入文件
2. **调用 ofep-converter**：使用 `Skill` 工具调用 `ofep-converter`，以方案设计文档作为输入
3. **等待转换完成**：ofep-converter 将自动完成内容映射、4+1 视图合规处理、格式合规检查，输出符合社区规范的 oFEP Markdown 文档
4. **保存 oFEP 文档**：将生成的 oFEP 文档保存到与设计文档相同的目录下，文件名按 `xxxx-ofep-<描述>.md` 规约（`<描述>` 与方案设计文件名一致，如方案设计为 `Mooncake-Local-Cache-design.md`，则 oFEP 为 `xxxx-ofep-Mooncake-Local-Cache.md`）

**oFEP 与原设计文档的关系**：

| 内容 | 原方案设计文档 | oFEP 提案 |
|------|--------------|-----------|
| 4+1 架构视图 | ✓ 保留 | ✓ 保留（PlantUML 合规） |
| DFX 质量属性 | ✓ 保留 | ✓ 保留 |
| 关键设计决策表 | ✓ 保留 | ✗ 不包含 |
| Story 分解/排期 | ✓ 保留 | ✗ 不包含 |
| 工程操作清单 | ✓ 保留 | ✗ 不包含 |
| 提案摘要/动机 | — | ✓ oFEP 规范章节 |
| 毕业标准 | — | ✓ oFEP 规范章节 |
| 生产可用性审查 | — | ✓ oFEP 规范章节 |
| 发布签核清单 | — | ✓ oFEP 规范章节 |

**注意事项**：
- oFEP 是**额外的输出件**，不替代原有的方案设计文档；两篇文档各有侧重，并行交付
- oFEP 面向社区提案/评审流程，侧重提案论述、架构视图和毕业标准；原设计文档面向工程实施，侧重 Story 分解与可执行操作清单
- 如用户明确表示不需要 oFEP 输出，可跳过此阶段

**阶段 5 交付物**：

## 交互策略总结


| 阶段  | 交互模式  | 要点                  |
| --- | ----- | ------------------- |
| 澄清  | 一问一答  | 每次一个问题，聚焦高影响决策      |
| 探索  | 呈现选项 + **停等** | 端到端方案或 **Q1…Qn 分项**；可批量一次答或逐项多轮；**全部 Q 决清前不得写正式设计文档** |
| 设计  | 分块交付  | 仅在选择确定后从块 1 开始；每块完成后征求意见再继续        |
| 评审  | 自评+确认 | 展示自评矩阵，请用户最终确认      |
| oFEP 转换 | 自动转换 | 设计文档定稿后调用 `ofep-converter`，额外生成社区提案文档；用户可跳过 |


**反模式**（必须避免）：

- 读完需求直接输出完整设计文档
- **阶段 2 已给出多方案与权衡，却在同一轮或未经用户回复就输出完整/大块设计文档**（含块 1～5、4+1 全套图、DFX/Story）
- **以 Agent 推荐代替用户决策**，未获明确回复即进入阶段 3
- **把多个正交决策塞进「方案 A/B」却不允许分项改选**，或**只决一条 Q 就当阶段 2 结束**
- 跳过方案比选直接给出唯一方案（除非已按上文说明「无实质备选」且用户已确认）
- 所有内容一次性输出不分块
- 未读取项目代码就做架构映射
- 未做 SOTA 调研就声称"业界最佳"
- **成稿中写入备选方案、比选矩阵、用户选型过程或未采纳路径详述**（应仅在对话中完成）
- **Story 未挂靠基线 SR/AR 或编号/表述与基线不一致**；缺少追溯表或操作清单流于空话
- **阶段 4 完成后未调用 ofep-converter 生成社区提案文档**（除非用户明确跳过）
