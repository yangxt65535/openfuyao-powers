# openFuyao Story 编码实现

基于方案设计文档实现特定 Story，采用双 Agent 协作模式确保代码质量。

## 工作流程概览

```
输入：Story 信息 + 方案设计文档
  │
  ▼
创建 .story-plan.md 记录计划
  │
  ├─ 步骤 1: 业务代码编写 ──────────────────────────┐
  │  │                                               │
  │  ▼                                               │
  │ Planner Agent (A): 规划实现 → 编写代码           │
  │  │                                               │
  │  ▼                                               │
  │ Reviewer Agent (B): 代码评审 → 反馈问题          │
  │  │                                               │
  │  └── 循环 ≤3 次 ─────────────────────────────────┘
  │       │
  │       ▼
  │ 业务代码完成
  │
  ├─ 步骤 2: 单元测试生成 ──────────────────────────┐
  │  │                                               │
  │  ▼                                               │
  │ Planner Agent (A): 规划测试 → 编写测试           │
  │  │                                               │
  │  ▼                                               │
  │ Reviewer Agent (B): 测试评审 → 反馈问题          │
  │  │                                               │
  │  └── 循环 ≤2 次 ─────────────────────────────────┘
  │       │
  │       ▼
  │ 单元测试完成
  │
  ▼
输出：完成的代码 + 测试文件
```

## 输入要求

用户需提供：
1. **Story 编号**：如 S1, S2 等
2. **Story 名称/标题**
3. **关联 SR/AR**：关联的系统需求和分配需求
4. **验收标准**：Story 的验收条件
5. **方案设计文档路径**（可选，如有详细设计文档）
6. **操作清单**：开发与验证操作清单（来自方案设计）

## 核心机制：双 Agent 协作

### Agent A - Planner（规划与实现）

**职责**：
- 理解 Story 需求和验收标准
- 制定实现计划（文件、函数、接口）
- 编写业务代码或单元测试
- 根据评审反馈修改代码

**约束**：
- 每次修改后必须自验证（语法、编译、基本逻辑）
- 必须响应所有评审意见
- 避免过度设计，保持代码简洁

### Agent B - Reviewer（评审与反馈）

**职责**：
- 审查代码质量、逻辑正确性、边界处理
- 检查是否符合项目规范和最佳实践
- 识别潜在问题和风险
- 给出具体、可执行的修改建议

**约束**：
- **避免钻牛角尖**：不追求代码完美，关注关键问题
- **聚焦重点**：逻辑错误、边界情况、安全风险、性能热点
- **忽略**：命名偏好、代码风格（已有 linter 处理）、微优化
- **建设性**：每个问题都应给出具体修改建议

## 迭代规则

| 步骤 | 最大迭代次数 | 终止条件 |
|------|-------------|---------|
| 业务代码 | 3 轮 | 评审无重大问题 或 达到最大次数 |
| 单元测试 | 2 轮 | 评审无重大问题 或 达到最大次数 |

**每轮流程**：
1. A 编写/修改代码
2. B 评审并输出评审意见
3. 如有问题 → A 修改 → 进入下一轮
4. 如无问题或达到最大次数 → 步骤完成

## 详细工作流

### 阶段 0：初始化与计划创建

1. **读取输入**
   - 获取 Story 信息（编号、名称、SR/AR、验收标准、操作清单）
   - 如有方案设计文档，读取相关内容

2. **创建计划文件**
   - 在项目根目录创建 `.story-plan.md`
   - 填充 Story 信息部分
   - 初始化业务代码和单元测试计划框架

### 阶段 1：业务代码编写

#### 第 1 轮：Planner 规划与实现

**Planner Agent 任务**：
1. 更新 `.story-plan.md`：标记为 `in_progress`
2. 分析需求：
   - 理解 Story 功能要求
   - 识别需要修改/创建的文件
   - 确定接口和依赖
3. 制定实现计划（写入 `.story-plan.md`）
4. 编写业务代码：
   - 按照操作清单实现功能
   - 遵循项目代码规范
   - 确保符合验收标准
5. 自验证：
   - 语法检查
   - 编译/构建检查
   - 基本逻辑验证
6. 更新 `.story-plan.md` 记录实现内容

#### 第 1 轮：Reviewer 评审

**Reviewer Agent 任务**：
1. 读取 Planner 编写的代码
2. 按照评审维度检查：
   - **逻辑正确性**：是否实现需求、边界处理、错误处理
   - **代码质量**：可读性、可维护性、复杂度
   - **安全风险**：注入、越界、并发问题
   - **性能**：热点路径、资源泄漏
   - **兼容性**：API 兼容、版本兼容
3. **约束检查**：
   - 不评论命名偏好（除非明显不当）
   - 不评论代码风格（已有 linter）
   - 不做微优化建议
   - 每类问题最多提 3 个最重要的
4. 输出评审意见：
   - 重大问题（必须修改）
   - 建议改进（可选修改）
   - 认可点（做得好的地方）
5. 更新 `.story-plan.md` 记录评审意见

#### 后续轮次

**如果存在问题**（且未达最大轮次）：
1. Planner 根据评审意见修改代码
2. 更新 `.story-plan.md` 记录修改内容
3. 进入下一轮评审

**如果没有重大问题或已达最大轮次**：
1. 标记业务代码为 `completed`
2. 进入阶段 2

### 阶段 2：单元测试生成

流程与阶段 1 相同，但关注点不同：

**Planner Agent 任务**：
1. 分析需要测试的功能点
2. 确定测试覆盖范围：
   - 正常路径
   - 边界条件
   - 错误处理
   - 并发场景（如适用）
3. 编写单元测试代码
4. 确保测试可运行（mock、依赖处理）

**Reviewer Agent 任务**：
1. 检查测试覆盖率（关键路径是否覆盖）
2. 检查测试有效性（断言是否合理）
3. 检查测试可维护性（是否过度耦合实现）
4. 识别测试盲点

**最大迭代次数**：2 轮

### 阶段 3：完成与验证

1. **更新计划文件**
   - 标记所有任务为 completed
   - 记录最终文件列表

2. **本地验证**（根据操作清单）
   - 运行项目构建命令
   - 运行单元测试
   - 确认无错误

3. **输出总结**
   - 实现的业务代码文件
   - 单元测试文件
   - 关键实现要点
   - 注意事项

## 评审维度参考

### Planner 自验证清单
- [ ] 代码能编译/解释通过
- [ ] 基本逻辑符合需求
- [ ] 无明显的语法错误
- [ ] 依赖已正确引入

### Reviewer 检查清单
- [ ] **逻辑正确性**
  - 是否正确实现了需求
  - 边界条件是否处理
  - 错误路径是否有处理
  - 并发场景是否安全
- [ ] **代码质量**
  - 函数职责是否单一
  - 参数校验是否充分
  - 返回值是否明确
  - 是否有重复代码
- [ ] **安全与风险**
  - 是否有注入风险
  - 是否有越界访问
  - 是否有资源泄漏
  - 是否有竞态条件
- [ ] **性能考虑**
  - 热点路径是否合理
  - 是否有明显的性能陷阱
  - 资源使用是否合理

**Reviewer 输出格式**：
```markdown
## 评审意见

### 重大问题（必须修改）
1. <问题描述> - <具体位置> - <修改建议>
2. ...

### 建议改进（可选）
1. <建议> - <理由>
2. ...

### 做得好的地方
- <认可点>

### 总体评价
<一句话总结>
```

## 质量门禁

- [ ] `.story-plan.md` 已创建并包含完整信息
- [ ] 业务代码至少经过 1 轮评审
- [ ] 单元测试至少经过 1 轮评审
- [ ] 所有代码文件已保存到磁盘
- [ ] 代码可以编译/运行
- [ ] 单元测试可以执行

## 使用示例

用户输入：
```
请实现 Story S3：Pod 状态同步优化

Story 信息：
- 编号: S3
- 名称: Pod 状态同步优化
- 关联 SR: SR2 - 容器状态实时感知能力
- 关联 AR: AR2.1 - Controller 状态同步逻辑优化
- 验收标准: 
  - AC2.1: Pod 状态变更延迟 < 1s
  - AC2.2: 状态同步失败有重试机制
- 操作清单: 
  - 修改 internal/controller/pod_controller.go
  - 添加重试逻辑
  - 添加单元测试
```

Skill 将：
1. 创建 `.story-plan.md`
2. Planner 编写业务代码
3. Reviewer 评审（多轮）
4. Planner 编写单元测试
5. Reviewer 评审（多轮）
6. 输出完成的代码
